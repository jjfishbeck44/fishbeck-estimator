# CLAUDE.md

## Project Overview

Fishbeck Estimator is an AI-powered construction cost estimator for Fishbeck Innovations. Users describe a project in plain text and receive a structured cost breakdown generated by Claude.

**Tech stack:** Vanilla JavaScript frontend, Node.js serverless backend (Vercel Functions), Claude API (Anthropic SDK), Upstash Redis for rate limiting.

## Architecture

```
Client (public/)  →  POST /api/estimate  →  api/estimate.js (handler)
                                                ├── lib/ratelimit.js  (Upstash Redis, 10 req/60s/IP)
                                                ├── lib/prompt.js     (system prompt + pricing data)
                                                └── lib/claude.js     (Claude API, 25s timeout)
```

**Request flow:** CORS check → rate limit → input validation (non-empty, ≤1000 chars) → Claude API call → JSON response.

**Frontend state machine:** INPUT → LOADING → RESULTS / CLARIFICATION / ERROR

## Directory Structure

```
api/
  estimate.js          # Vercel serverless POST handler (orchestrator)
lib/
  claude.js            # Claude API wrapper (claude-sonnet-4-6, Promise.race timeout)
  prompt.js            # System prompt with pricing reference data (pure data, no I/O)
  ratelimit.js         # Upstash sliding-window rate limiter (fails open)
public/
  index.html           # Single-page app (form, results table, loading states)
  css/style.css        # Custom design system (navy #1B3A5C, gold #C8963E)
  js/estimator.js      # Frontend logic (IIFE, state machine, fetch API)
tests/
  estimate.test.js     # Handler tests (mocked deps)
  claude.test.js       # Claude wrapper tests (mocked SDK)
  prompt.test.js       # System prompt content validation
  ratelimit.test.js    # Rate limiter tests (fail-open behavior)
```

## Testing Conventions

- **Framework:** Jest v29, Node environment
- **Pattern:** All tests in `tests/**/*.test.js`
- **Mocking:** All external dependencies are mocked (`@anthropic-ai/sdk`, Upstash). No live API calls in tests.
- **Helpers:** Tests use `makeReq()` / `makeRes()` factory functions for request/response objects
- **Coverage areas:** Happy paths, error states, rate limiting, input validation, timeout handling

## Code Conventions

### General
- **Plain JavaScript** — no TypeScript, no transpilation
- **CommonJS** (`require`/`module.exports`) on the backend
- **IIFE pattern** with `'use strict'` on the frontend
- **No linter or formatter configured** — match the existing code style
- **No semicolons enforced** — mixed usage, follow surrounding code

### Naming
- `camelCase` for functions and variables
- `UPPER_SNAKE_CASE` for constants (`STATES`, `TIMEOUT_MS`, `MAX_INPUT_LENGTH`)
- Descriptive file names matching their single responsibility

### Commit Messages
- Conventional commits: `feat:`, `fix:`, `docs:`, `chore:`
- Descriptive but concise (e.g., `feat: add Claude API wrapper with 25s timeout and JSON parsing`)

## Key Patterns

- **Fail-open rate limiting:** If Upstash is misconfigured, requests are allowed through so the estimator keeps working during setup.
- **25s timeout via `Promise.race`:** Stays within Vercel's 30s function limit. Throws typed `api_timeout` error.
- **Typed error codes in API responses:** `rate_limited`, `missing_input`, `input_too_long`, `api_timeout`, `invalid_json`, `internal_error`.
- **XSS prevention:** Frontend uses `escHtml()` to sanitize all user-provided text before rendering.
- **JSON-only Claude responses:** System prompt instructs Claude to return structured JSON with `status`, `line_items`, `totals`, `notes`, and `out_of_scope` fields.
